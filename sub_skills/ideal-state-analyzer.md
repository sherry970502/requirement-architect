# 项目理想化状态分析（Ideal State Analyzer）

这是一套通过多维度系统性发散，提炼项目理想化状态的分析方法。核心逻辑是：**找到项目理想的、完美的、终极的状态，如北极星一般，引领项目的规划及后续工作的开展**。

分析分为 20 个步骤，每个步骤从不同维度发散理想化状态，最终输出一份按重要度排序的理想化状态清单。

## 输入要求

用户需提供以下项目基础信息：

1. **项目标题**（必填）
2. **项目描述**（必填）
3. **目标用户**（必填）
4. **核心价值**（必填）
5. **相关会议纪要**（可选，如知识库未对接则建议提供）

## Step 1：项目信息回顾

通过项目的标题、描述、目标用户和核心价值，对项目信息进行整体回顾。

**目的：**
- 建立对项目的基础认知
- 明确当前已知的关键信息
- 为后续发散提供基准

**输出：**

```
项目信息回顾：

项目标题：[项目名称]
项目描述：[简要描述]
目标用户：[用户群体]
核心价值：[核心价值陈述]

关键信息提炼：
  - [从以上信息中提炼的 3-5 个关键点]
```

## Step 2：历史会议纪要提炼

通过数据接口或用户提供的会议纪要，提炼与该项目相关的理想化状态。

**提炼原则：**
- 寻找讨论中关于"理想状态""希望达到""最好能"等表述
- 提取重要决策背后隐含的价值导向
- 识别反复强调的期待或愿景

**输出：**

```
历史会议纪要提炼：

理想化状态 1：[描述]
  来源：[哪次会议/哪段讨论]
  
理想化状态 2：[描述]
  来源：...

（提炼所有相关的理想化状态）
```

## Step 3：行业标准参考

从成熟优秀的行业标准中寻找启发。

**参考来源：**
- 行业权威奖项（如红点设计奖、艾美奖、金摇杆奖等）
- 行业规范与标准（如 ISO 标准、行业白皮书）
- 成熟的解决方案（如行业标杆产品）

**输出：**

```
行业标准参考：

参考 1：[行业标准/奖项/标杆产品]
  启发的理想化状态：[描述]
  
参考 2：...
```

## Step 4：典型情景发散

针对最重要的干系人和目标用户，选取广泛且高频的场景进行发散。

**场景选择标准：**
- 覆盖面广（大部分用户都会遇到）
- 使用频率高（经常发生）
- 价值密度大（对用户/干系人影响显著）

**输出：**

```
典型情景发散：

场景 1：[场景描述]
  理想化状态：[在这个场景下的理想体验]
  
场景 2：...

（至少 5-8 个典型场景）
```

## Step 5：竞品分析启发

从三个层次分析竞品，寻找理想化状态。

**分析层次：**
1. **功能相同**：直接竞品，做同样的事
2. **核心功能相似**：间接竞品，用不同方式解决类似问题
3. **本质完全相同**：深层竞品，满足同样的底层需求

**输出：**

```
竞品分析启发：

竞品 1：[产品名称]（功能相同）
  值得学习的理想化状态：[描述]
  
竞品 2：[产品名称]（核心功能相似）
  值得学习的理想化状态：[描述]
  
竞品 3：[产品名称]（本质完全相同）
  值得学习的理想化状态：[描述]
```

## Step 20：理想化状态复盘

对以上发散的所有维度的理想化状态做复盘、整理和筛选，按重要度排列。

**整理原则：**
- 去重（多个维度提到的相同理想化状态合并）
- 提炼（将相似的表述整合成更清晰的描述）
- 评估（判断每条理想化状态的重要度）

**重要度评级标准：**
- **P0（核心必备）**：不实现这个，项目失去灵魂
- **P1（重要提升）**：实现后显著提升项目价值
- **P2（锦上添花）**：实现后增加亮点，但非关键

**输出格式：**

```markdown
# 项目理想化状态清单（按重要度排序）

## P0 级 - 核心必备

| 序号 | 理想化状态描述 | 来源维度 | 重要度评级 |
|------|---------------|---------|-----------|
| 1 | [清晰描述] | [维度名称] | P0 |
| 2 | [清晰描述] | [维度名称] | P0 |
| ... | ... | ... | ... |

## P1 级 - 重要提升

| 序号 | 理想化状态描述 | 来源维度 | 重要度评级 |
|------|---------------|---------|-----------|
| X | [清晰描述] | [维度名称] | P1 |
| ... | ... | ... | ... |

## P2 级 - 锦上添花

| 序号 | 理想化状态描述 | 来源维度 | 重要度评级 |
|------|---------------|---------|-----------|
| Y | [清晰描述] | [维度名称] | P2 |
| ... | ... | ... | ... |
```

## 使用说明

### 输入要求

用户需提供：
1. 项目标题（必填）
2. 项目描述（必填）
3. 目标用户（必填）
4. 核心价值（必填）
5. 相关会议纪要（可选，建议提供）

### 质量原则

- **全面发散**：不能跳过任何一个维度，每个维度都要认真思考
- **具体明确**：理想化状态要具体到可以验证，不能是空泛表述
- **北极星导向**：每条理想化状态都应该是"最理想""最完美"的状态，而非"可以做到"的状态
- **重要度准确**：P0/P1/P2 的划分要基于对项目的深刻理解，不能随意评级
- **去重精炼**：Step 20 的整理要确保没有重复，表述要简洁有力

### 进阶用法

**场景 1：快速验证某个维度**
如果只想探索某个特定维度，可以明确：「只做 Step 6 AI 维度联想」

**场景 2：深化核心维度**
可以要求某些维度给出更多理想化状态：「Step 10 游戏化元素至少给 10 个理想化状态」

**场景 3：迭代优化**
可以基于已有的理想化状态清单，要求重新评级：「重新评估这些理想化状态的重要度」
