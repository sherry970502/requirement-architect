# 使用场景分析框架(Scenario Analyzer)

这是一套从目标用户出发,系统性还原真实使用情境的分析方法。核心逻辑是:**基于目标用户,横向展开其接触产品的所有主要场景,针对每个场景做结构化深度分析(时空、触发、动作、情绪、障碍、期待),最终按重要度排序产出可指导设计的场景清单**。

分析分为 6 个步骤,针对每个核心场景输出完整的场景画像(时空、触发、动作流、情绪曲线、障碍与期待)。

---

## 核心概念定义

### 什么是使用场景?

**使用场景**是指目标用户在真实生活中接触、使用或受益于产品的具体情境。它不是抽象的"用户需求",而是**具体到时间、地点、情绪、动作**的一段真实情节。

### 使用场景的完整要素

一个完整的使用场景至少包含以下要素:

```
【时空】什么时间 + 什么地点
【触发】为什么会开始使用
【动作】具体做了哪些操作
【情绪】过程中的心理状态变化
【障碍】遇到的困难或摩擦
【期待】希望获得什么结果
```

**示例:**

- ✅ 正确的场景描述:"周日晚上 9 点,妈妈把孩子哄睡后,坐在客厅沙发上打开 App,想在睡前看 15 分钟课程充实自己,但担心声音吵醒孩子,一边戴耳机一边刷——希望找到能让她'静静看一会儿'的内容"
- ❌ 错误的场景描述:"妈妈在家里使用产品学习"(太抽象,缺要素)

### 场景与"用户画像中典型情景"的区别

用户画像里的典型情景是"给一个具体用户写一个故事",使用场景分析是"为一类需求横向梳理所有主要情境"。前者是人物剧,后者是场景地图。两者互补。

## 输入要求

用户需提供:

- **项目/产品需求描述**(必填):说明要做什么产品或项目
- **目标用户**(必填):来自 `target-user-analyzer` 的产出,或用户手动提供
- **核心价值**(可选但推荐):来自 `core-value-structure` 的产出,用于校准场景的价值落点

## Step 1:场景横向发散

基于目标用户,穷举该用户接触/使用/脱离产品的所有主要场景,至少覆盖以下 7 个维度。

### 1.1 从时间维度发散

**思考角度:**
- 一天中的典型时段:清晨、上班通勤、工作间隙、午休、下班路上、晚饭后、睡前
- 一周中的典型时点:工作日 vs 周末
- 一年中的典型节点:节假日、季节转换、重要人生节点
- 产品生命周期:初次接触、探索期、习惯期、流失期、回流期

### 1.2 从空间维度发散

**思考角度:**
- 家中(卧室/客厅/厨房/卫生间)
- 工作场所(办公室/会议室/通勤路上)
- 公共空间(咖啡馆/书店/商场/地铁)
- 特殊空间(学校/医院/旅途/户外)

### 1.3 从情绪维度发散

**思考角度:**
- 积极情绪场景:兴奋、期待、好奇、愉悦时
- 消极情绪场景:焦虑、疲惫、沮丧、无聊时
- 中性情绪场景:日常、习惯性、例行公事时
- 特殊情绪场景:需要安慰、需要激励、需要宣泄时

### 1.4 从社交维度发散

**思考角度:**
- 独自一人时
- 与家人在一起(伴侣/父母/孩子)
- 与朋友在一起
- 在陌生人环境中(公共场合)
- 在特定社群中(同事/学习小组/兴趣圈)

### 1.5 从任务/目的维度发散

**思考角度:**
- 明确任务驱动(要完成某件事)
- 信息获取(想了解什么)
- 情绪调节(想放松/想充实)
- 社交驱动(想被看到/想连接)
- 打发时间(无明确目的)

### 1.6 从设备/渠道维度发散

**思考角度:**
- 手机端:锁屏、通知、打开、切换
- PC 端:工作间隙、搜索、深度使用
- 跨端场景:手机到 PC 的流转、碎片到完整的过渡
- 线下场景:实体接触、口碑传播、物料触达

### 1.7 从生命周期维度发散

**思考角度:**
- 首次接触:怎么知道产品的?第一次使用时的场景?
- 探索期:在什么情境下反复尝试?
- 习惯期:什么时候会自然想起来用?
- 流失期:什么情境下决定不再用?
- 回流期:什么触发让用户重新想起?

**输出:**

```
场景横向发散:

维度 1:时间
  场景 1.1:[具体场景描述,一句话]
  场景 1.2:...

维度 2:空间
  场景 2.1:...
  ...

维度 3:情绪
  ...

(每个维度至少 3-5 个场景,7 个维度合计 25-40 个场景候选)
```

## Step 2:场景归并与核心场景识别

对横向发散出的大量场景候选进行归并、去重、聚类,识别出**核心使用场景**。

### 2.1 归并原则

**去重:** 不同维度下出现的相似场景合并(例如"睡前在卧室用手机"可能同时被"时间"和"空间"维度提到)

**聚类:** 将本质相同的场景聚为一类(例如"早上通勤地铁上""晚上通勤地铁上"可以合并为"通勤场景"——如果两者使用模式基本一致)

**提炼:** 将过于具象的场景抽象到可识别的层次(例如"周六下午 3 点在星巴克看书"→"周末在咖啡馆的轻度学习时段")

### 2.2 核心场景筛选标准

不是所有场景都值得深度分析。按以下标准筛选出 **3-7 个核心场景**:

**筛选标准:**

- **覆盖度**:有多少比例的目标用户会遇到?
- **频率**:单个用户会多频繁遇到这个场景?
- **价值密度**:在这个场景下,产品提供的价值是否特别高?
- **差异性**:这个场景和其他场景相比是否有本质不同(不同的动作/情绪/障碍)?
- **设计驱动力**:分析这个场景能否带来独特的设计启发?

**筛选原则:**

- 高覆盖 + 高频率 的场景必选
- 高价值密度但低频率的场景可选(关键时刻)
- 高差异性的场景可选(有独特启发)
- 低覆盖 + 低频率 + 无差异性的场景剔除

**输出:**

```
核心场景清单(3-7 个):

核心场景 1:[场景名称]
  一句话定位:[时间 + 地点 + 触发 + 目的]
  来源维度:[发散时来自哪几个维度的聚合]
  筛选理由:[为什么是核心]

核心场景 2:[场景名称]
  ...

(通常 3-7 个,过少覆盖不全,过多失去重点)
```

## Step 3:单场景深度画像

针对每个核心场景,按以下结构做完整画像。

### 3.1 时空要素

**思考角度:**
- **时间**:具体时段、持续时长、是否被打断
- **地点**:物理空间特征、光线、声音、其他在场物/人
- **氛围**:这个时空带来的整体氛围是什么样的

### 3.2 触发要素

**思考角度:**
- **外部触发**:收到通知、看到物料、他人提起、环境变化
- **内部触发**:想到什么、感受到什么、产生了什么需要
- **触发链**:从"什么都没发生"到"决定使用产品",中间经历了什么

### 3.3 动作要素(动作流)

**思考角度:**
- 用户从开始到结束,**依次**做了哪些具体动作?
- 每个动作平均花多少时间?
- 哪些动作是"主线动作"(推进目标),哪些是"辅线动作"(过渡、试错)?
- 最常见的动作路径是什么?有没有常见的岔路?

### 3.4 情绪要素(情绪曲线)

**思考角度:**
- 开始前的情绪基调是什么?
- 过程中情绪如何起伏?
- 在哪些动作节点情绪会产生显著变化(上扬/下降)?
- 结束时的情绪是什么?(满足 / 遗憾 / 中性 / 被激起下一次)

### 3.5 障碍要素

**思考角度:**
- **客观障碍**:信号差、电量低、光线不好、没有耳机、被打扰
- **主观障碍**:不会用、看不懂、不好意思、怕麻烦、不信任
- **流程障碍**:步骤太多、等待时间长、需要切换、需要登录
- **内容障碍**:不感兴趣、不符合期待、找不到想要的

### 3.6 期待要素

**思考角度:**
- 功能期待:希望产品能帮我完成什么?
- 体验期待:希望这个过程感觉怎么样?
- 情绪期待:希望结束时获得什么情绪?
- 隐性期待:用户自己可能没意识到、但如果被满足会惊喜的期待

**输出(针对每个核心场景):**

```
# 核心场景 X:[场景名称]

## 时空
- 时间:[具体时段和时长]
- 地点:[物理空间]
- 氛围:[氛围描述]

## 触发
- 外部触发:[具体触发]
- 内部触发:[具体触发]
- 触发链:[从 0 到 1 的过程]

## 动作流
1. [动作 1](耗时 ~X 秒)
2. [动作 2](耗时 ~X 秒)
3. ...
主线动作:[哪几步]
常见岔路:[哪几步]

## 情绪曲线
- 起始情绪:[描述]
- 过程关键节点情绪变化:
  - 动作 X 节点:[情绪变化]
  - 动作 Y 节点:[情绪变化]
- 结束情绪:[描述]

## 障碍
- 客观障碍:[列举]
- 主观障碍:[列举]
- 流程障碍:[列举]
- 内容障碍:[列举]

## 期待
- 功能期待:[描述]
- 体验期待:[描述]
- 情绪期待:[描述]
- 隐性期待:[描述]
```

## Step 4:场景间关系与流转分析

单场景分析完成后,分析场景之间的**关系和流转**——这是场景地图区别于单点分析的关键。

### 4.1 场景间的关系类型

**并列关系:** 两个场景独立存在,用户可能只经历其中之一
**承接关系:** 场景 A 之后经常发生场景 B(A 触发 B)
**替代关系:** 场景 A 和场景 B 满足同一需求,用户会二选一
**互补关系:** 场景 A 和场景 B 共同组成完整体验,缺一不可
**演化关系:** 用户随着使用深入,会从场景 A 逐渐转向场景 B(新手 → 老用户)

### 4.2 场景流转图

将识别出的核心场景,用流转关系连接起来,画出用户在场景间的"移动路径"。

**输出:**

```
场景流转关系:

承接流转:
  场景 A → 场景 B(X% 用户会在 A 后进入 B)
  场景 B → 场景 C(...)

替代选择:
  场景 D ⟷ 场景 E(用户二选一)

演化路径:
  新手期:主要在场景 F
  成长期:逐渐转向场景 G
  成熟期:以场景 H 为主

关键流转节点:
  - [节点 1]:在这里用户决定是否继续深度使用
  - [节点 2]:在这里用户决定是否流失
```

## Step 5:关键时刻识别(Moments of Truth)

在所有核心场景和流转节点中,识别"关键时刻"——决定用户是否留下、是否推荐、是否付费的那几个瞬间。

### 5.1 关键时刻的类型

**入门时刻:** 用户首次使用产品的前 3 分钟——决定是否继续用
**顿悟时刻(Aha Moment):** 用户第一次真正感受到产品价值的瞬间——决定是否成为忠实用户
**信任时刻:** 产品出错或用户遇到困难时——决定用户对产品的信任度
**分享时刻:** 用户产生"想推荐给别人"冲动的瞬间——决定产品的口碑传播
**付费时刻:** 用户决定为产品付钱的瞬间——决定商业模式成立

### 5.2 识别标准

不是所有产品都有全部 5 种关键时刻。根据项目性质识别 **2-4 个** 对本项目最关键的时刻,并说明:

- 这个关键时刻在哪个场景中发生?
- 这个时刻的成败由什么决定?
- 如何设计才能最大化成功率?

**输出:**

```
关键时刻清单:

关键时刻 1:[名称,如"首次使用的 Aha Moment"]
  发生场景:[核心场景 X]
  发生节点:[动作流的第 Y 步]
  成败关键:[决定成败的因素]
  设计启发:[如何最大化成功率]

关键时刻 2:...
```

## Step 6:场景优先级排序

对所有核心场景进行排序,输出最终的场景清单。

### 排序标准

- **覆盖度 × 频率**:场景的体量(有多少用户会多频繁地经历)
- **价值密度**:在该场景下,用户获得的价值有多大
- **设计驱动力**:该场景对产品设计的指导性有多强
- **关键时刻关联度**:该场景是否承载关键时刻

### 优先级分级

- **P0(核心场景)**:高覆盖 + 高频率 + 承载核心价值 → 产品设计必须优先满足
- **P1(重要场景)**:部分维度较高 → 设计要覆盖到,但优先级次于 P0
- **P2(辅助场景)**:较低覆盖或频率,但有独特价值 → 锦上添花

**输出:**

```markdown
# 使用场景清单(按优先级排序)

## P0 级 - 核心场景

| 序号 | 场景名称 | 一句话定位 | 关联关键时刻 | 优先级 |
|------|---------|-----------|-------------|--------|
| 1 | [场景] | [定位] | [关键时刻] | P0 |
| 2 | ... | ... | ... | P0 |

## P1 级 - 重要场景

| 序号 | 场景名称 | 一句话定位 | 关联关键时刻 | 优先级 |
|------|---------|-----------|-------------|--------|
| X | ... | ... | ... | P1 |

## P2 级 - 辅助场景

| 序号 | 场景名称 | 一句话定位 | 关联关键时刻 | 优先级 |
|------|---------|-----------|-------------|--------|
| Y | ... | ... | ... | P2 |
```

## 3. 单场景深度画像(示例:哄睡后的独处时光)

### 时空
- 时间:周日晚上 9:00 - 10:00,持续约 30-60 分钟
- 地点:卧室床上 或 客厅沙发,光线昏暗(避免吵醒孩子)
- 氛围:安静、松弛、略带疲惫但期待"属于自己的时间"

### 触发
- 外部触发:孩子终于睡着,确认呼吸平稳
- 内部触发:"今天终于熬到头了""我也想学点东西了""不想又刷手机刷到睡着"
- 触发链:哄睡 → 轻手轻脚离开 → 坐下 → 长舒一口气 → 想做点有意义的事 → 打开 App

### 动作流
1. 打开手机,习惯性检查消息(~30 秒)
2. 打开 App(~5 秒)
3. 浏览首页推荐(~1-2 分钟)
4. 选一节课打开(~10 秒)
5. 戴上耳机(~15 秒)
6. 听课/看课(10-20 分钟)
7. 课程结束 → 决定再看一节 或 关掉睡觉(~30 秒决策)

主线动作:选课 → 听课
常见岔路:第 3 步可能花 5 分钟选不出想看的 → 刷其他 App → 放弃学习

### 情绪曲线
- 起始情绪:疲惫但放松,有"终于自由了"的释放感
- 过程关键节点:
  - 第 3 步(选课):如果选不出会有挫败感,开始怀疑"要不还是刷刷视频算了"
  - 第 6 步(听课中):如果内容契合,会进入"浸润"状态,感到充实
  - 第 7 步(结束):如果内容好,有满足感;如果平平,有空虚感
- 结束情绪:理想状态下是"今天没白过"的满足,最糟是"又浪费了一晚上"

### 障碍
- 客观障碍:没找耳机就没法看有声课;光太亮会不舒服
- 主观障碍:疲惫状态下不想思考,选课困难
- 流程障碍:每次打开要从头选,没有"继续上次"的顺滑入口
- 内容障碍:大部分课程都太长,听不完就要睡觉,有挫败感

### 期待
- 功能期待:希望能快速找到 10-15 分钟能听完的内容
- 体验期待:希望界面安静不刺眼,声音柔和不突兀
- 情绪期待:希望结束时有"今晚学到了"的满足感
- 隐性期待:如果 App 能像朋友一样"懂我现在的状态"、知道我不想思考只想听,那会很惊喜

## 使用说明

### 输入要求

用户需提供:
1. 项目/产品需求描述(必填)
2. 目标用户(必填,建议来自 target-user-analyzer 产出)
3. 核心价值(可选,建议来自 core-value-structure 产出)

### 质量原则

- **具体而非抽象**:所有场景描述必须有具体的时间、地点、动作,不能只写"用户在使用产品"
- **真实而非理想**:场景描述要还原真实会发生的情况,包括障碍和负面情绪,不能美化
- **完整要素**:每个核心场景都必须覆盖"时空/触发/动作/情绪/障碍/期待"六个要素,缺一不可
- **场景间有关系**:不能把场景当作孤立点,必须分析场景间的流转和关联
- **关键时刻有锚点**:关键时刻必须和具体的场景节点绑定,不能凭空讨论

### 调研工具使用

本 Skill 可自动调用 web_search 工具获取:
- 目标用户群体的典型生活作息和行为数据
- 相关品类产品的用户使用行为研究
- 特定场景下的用户心理研究

### 进阶用法

**场景 1:基于已有用户画像生成场景**
如果已经有 `target-user-analyzer` 的产出,可以直接输入:"基于以下用户画像,输出使用场景分析",AI 会复用画像细节。

**场景 2:深化某个核心场景**
如果只想深度分析某一个场景:"只分析'哄睡后的独处时光'这一个场景,做到最深"

**场景 3:场景驱动的设计复盘**
可以将已有产品映射到场景清单上:"这是我现在的产品功能,请评估它对每个核心场景的覆盖度"

**场景 4:新场景探索**
"基于现有场景清单,帮我探索 3 个我可能忽略但有价值的新场景"
